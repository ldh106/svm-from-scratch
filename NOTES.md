# 검증 노트

포트폴리오 정리를 위해 2026년 7월, 업로드 당시(2025년 12월) 코드를 다시
실행하여 보고서 수치와 대조 검증한 기록입니다.

## ✅ 일치 확인된 항목

`train_set.csv`(80개, 과목 제공 데이터)로 다시 학습시킨 결과, 아래 값들은
보고서와 정확히 일치했습니다.

- 학습 정확도: 93.75%
- 혼동행렬: [[40, 1], [4, 35]]
- w: [-2.1806347281497596, -0.8220265732112827, 0.6994535063667132]
- b: -0.34264903306975925
- test_set.csv(20개) 예측 결과: 0-11번 전환(1), 12-17번 비전환(0),
  18-19번 전환(1)

## ⚠️ 불일치 발견: 서포트 벡터 개수

보고서 6장(실험 결과 및 분석)에는 다음과 같이 기재되어 있습니다.

> "학습 완료 후 라그랑주 승수(α)를 분석한 결과, α > 0인 값을 가져 결정
> 경계 형성에 직접적으로 기여하는 서포트 벡터의 개수는 총 15개로 확인되었다.
> 이는 전체 훈련 데이터(80개)의 18.75%에 해당한다."

하지만 재실행한 모델의 `alphas` 배열을 직접 확인한 결과, 0보다 큰 α 값을
가진 데이터는 **23개(28.75%)**였습니다. threshold를 1e-6, 1e-4, 1e-3,
0.01, 0.05, 0.1로 바꿔가며 세어봐도 23개로 동일했고(threshold=0.5에서만
21개로 소폭 감소), 15개가 나오는 조건은 찾지 못했습니다.

세부적으로 보면:
- α = C(=1.0)에 도달한 "bound" 서포트 벡터: 19개
- 0 < α < C인 "margin" 서포트 벡터: 4개 (α ≈ 0.802, 0.619, 0.226, 0.194)
- 나머지는 α ≈ 0 (사실상 0)

모델의 다른 모든 출력값(정확도, 혼동행렬, w, b, test 예측)이 보고서와
정확히 일치하는 것으로 보아, **학습된 모델 자체는 동일**하다고 판단됩니다.
따라서 이 불일치는 코드 로직 문제라기보다, 팀원들이 보고서를 작성하는
과정에서 서포트 벡터 개수를 별도로 (아마 수동으로, 혹은 다른
threshold/스크립트로) 계산하며 생긴 차이로 추정됩니다. 정확한 원인은
재현하지 못했습니다.

## 참고: 실행 환경에서 겪었던 문제 (제출 당시)

- **CSV BOM 문제**: Windows/엑셀에서 저장한 CSV 파일 첫 컬럼명 앞에 보이지
  않는 BOM 문자(`\ufeff`)가 붙어 `KeyError: 'response_time_min'`가 발생함.
  `open()`을 `encoding="utf-8-sig"`로 바꿔서 해결.
- **테스트셋 재설계**: 처음 만든 test_set.csv 20개가 전부 "전환(1)"으로만
  예측되어 보고서에 넣기 부자연스러워, 학습된 w의 부호(응답시간↓·견적↓·
  상담횟수↑ → 전환↑)를 기준으로 0/1이 섞이고 경계 근처 케이스도 포함되도록
  값을 다시 설계함. 현재 레포의 `test_set.csv`가 이 재설계된 버전.

## 실험 범위에 대한 정직한 기록

보고서에는 "C=1.0으로 설정하여 학습을 진행하였다"고만 명시되어 있고,
**C 값 변화 실험이나 RBF 등 비선형 커널 실험은 실제로 진행하지 않았습니다.**
구현 과정에서 이런 실험을 추가하면 좋겠다는 논의는 있었지만, 실제 제출한
프로젝트는 선형 커널 + C=1.0 고정 조건에서의 결과만 포함합니다.
